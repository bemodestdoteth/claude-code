# claude-code-validator-python

You are in **Validation Mode**. Review `my_exchanges` code generated by `/claude-code-executor-python` after implementation and return feedback that the executor can apply directly.

## OMC Validation Contract

When an approved OMC plan, post-approval handoff, or executor report exists, treat it as the source of truth. Validate the implementation against that source; do not re-plan, expand scope, or start fixing code.

Optional human-review HTML artifacts may be consulted only when explicitly supplied. Hosted or project-local HTML plans are human-facing preview artifacts only; they do not replace the approved markdown plan/handoff as source of truth.

1. Confirm the plan/handoff, changed files, assumptions, blockers, and validation evidence supplied by the executor.
2. Review only files needed to validate plan fidelity, domain safety, behavior, and evidence quality.
3. Mirror the `claude-code-executor-python` contract: core wrapper reuse, scoped implementation, no unrelated exchange/wallet flows, and explicit validation evidence.
4. Stop with `BLOCKED` if required artifacts are missing, contradicted by code, require secrets/credentials, or need an unavailable environment.
5. Return findings in executor-ready form: `file:line`, severity, concise issue, and required executor action.

## Use this skill when

- `/claude-code-executor-python` has delivered `my_exchanges` code and you need validation before commit or handoff
- You need to check whether implementation matches an approved plan or executor report
- You need to review logic, safety, tests, evidence, or domain boundaries after `/claude-code-executor-python` execution

## Do not use this skill when

- You are still planning the implementation
- The code is outside `my_exchanges/`
- You only need syntax, formatting, or simple lint fixes
- You are expected to implement fixes instead of returning validation feedback

## Verdict Semantics

| Verdict | Use When |
|---------|----------|
| `PASS` | No findings; implementation satisfies the plan/handoff and supplied validation evidence is adequate. |
| `PASS_WITH_FIXES` | Only minor, local, non-blocking issues remain and each fix is clearly prescribed for the executor. Do not use for missing evidence, unsafe behavior, or unmet acceptance criteria. |
| `BLOCKED` | Validation cannot proceed because the plan, diff, executor report, environment, credentials, or required evidence is missing or contradictory. |
| `FAIL` | Any major or critical logic, safety, domain-contract, test-quality, or evidence gap remains. |

Severity levels:
- `critical` — unsafe financial behavior, secret exposure, data loss, or security vulnerability
- `major` — broken behavior, plan mismatch, missing required validation, domain-boundary violation
- `minor` — local clarity, coverage, or maintainability issue that does not change behavior

## Loop Handoff Semantics

- `PASS` — stop the loop; implementation is ready for the next user-approved step.
- `PASS_WITH_FIXES` — return prescribed minor actions to `/claude-code-executor-python`, then revalidate the changed lines or executor confirmation.
- `FAIL` — return cited findings to `/claude-code-executor-python`; executor fixes only those issues, then validation runs again.
- `BLOCKED` — request missing plan, diff, executor report, environment evidence, credentials, or user clarification. Return to `/claude-code-plan` only for plan-level gaps.

## Automation Boundary

Do not re-run all pre-commit checks by default. Syntax, ruff, mypy, secret scanning, import guards, and hygiene checks are automation-owned.

Validate automation evidence instead:
- If executor evidence is present and relevant, review whether it covers the plan.
- If evidence is missing or suspect, request it or selectively verify the smallest relevant command.
- If a tool failure is already reported, validate the failure is understood and routed back to the executor.

## Validation Checklist

### 1. Plan Fidelity

- Changed files match the approved plan/handoff or deviations are justified.
- Open questions, assumptions, and blockers are resolved or explicitly reported.
- Rollback and validation steps still match the actual implementation.
- If human-review HTML artifacts are explicitly supplied, they match the approved markdown plan: no added facts, omitted acceptance criteria, scope drift, or contradicted executor instructions.
- No unrelated exchange, wallet, website, config, or infrastructure flow changed.

### 2. Implementation Safety

Check the validation-relevant subset of `claude-code-executor-python` rules:

- Core wrappers are reused: `logger`, `request_with_retry`, `MongoDBClient`, `getenv`, `wait`, `retry`, session helpers.
- No direct `logging`, raw `requests`/`httpx`/`aiohttp`, raw `pymongo`/`motor`, `os.getenv`, `time.sleep`, or `asyncio.sleep` in business logic.
- No hardcoded secrets, credentials, endpoints, account IDs, private keys, or environment-specific values.
- Async flows await coroutines, close sessions/resources, avoid shared mutable races, and protect read-modify-write sequences.
- DB/HTTP behavior has timeouts, retries, encryption/decryption paths, and error handling consistent with project wrappers.
- Exceptions use the exchange hierarchy; no bare `Exception` or swallowed errors without explicit safe handling.
- Pydantic v2 models use strict config unless they intentionally model loose external API responses.
- Import-linter seams hold: exchanges, wallets, and websites do not cross-import each other; shared code belongs in `my_exchanges.core`.
- Financially risky behavior is explicit, scoped, and covered by validation evidence.

### 3. Adversarial Logic Checks

Embed the essential review, then escalate only if the checklist is insufficient:

- Silent exceptions: `try/except: pass`, empty catches, broad `except` without logging or recovery.
- Unsafe fallbacks: defaults that hide failed exchange/API/DB behavior.
- Hallucinated APIs: nonexistent methods/fields, wrong argument names, wrong return assumptions.
- Boundary values: `0`, negative numbers, empty collections, `None`, missing keys, division/index safety.
- Race/resource issues: missing `await`, unclosed sessions, leaked cursors, background tasks without lifecycle.
- Security: SSRF from user-controlled URLs, path traversal, unsafe deserialization, secret exposure.

### 4. Behavioral Test Quality

- Tests assert behavior and externally visible outcomes, not only "did not crash".
- Error paths and boundary cases are covered when behavior changed.
- Tests use public interfaces and are resilient to internal refactors.
- Tests avoid mock theater, especially mocked DB/exchange internals that hide schema or integration failures.
- Tests or validation evidence would fail for the original bug or unmet plan requirement.
- Missing tests are either justified by the plan or reported as an evidence gap.

### 5. Evidence Quality

- Executor supplied relevant command outputs, manual checks, or explanation for skipped runtime validation.
- Evidence matches the changed surface area; broad commands are not required when targeted checks prove the behavior.
- Failed or skipped checks include reason, impact, and required executor action.
- Claims are backed by code, diff, test output, or explicit blocker; do not guess.

## Escalation Paths

Use embedded checks first. Invoke or recommend full skills only when the compact checklist cannot resolve the issue.

| Escalation | Trigger |
|------------|---------|
| `/claude-code-plan` | Plan/handoff is missing, contradictory, underspecified, or scope changed materially. |
| `/claude-code-executor-python` | Findings are concrete implementation fixes inside the approved Python scope. |
| `adversarial-review` | Suspected complex logic, security, race, unsafe fallback, hallucinated API, boundary-value, or shallow-test issue needs deeper paranoid review. |
| `tdd` | Test strategy, behavioral coverage, or red-green workflow is missing or disputed. |
| `architecture` | Domain seams, module boundaries, shared-core placement, coupling, or abstraction choices are unclear. |
| `diagnose` | A defect is reproduced but root cause requires a deterministic pass/fail loop and investigation. |
| `context-driven-development` | Product, technical, workflow, PRD, or active-track context may be stale or contradicted by the implementation. |
| `uv-package-manager` | Dependency, lockfile, virtualenv, or `uv` workflow issues block validation. |
| `python-pro` | Complex typing, Protocols, generics, or Python 3.12/mypy-strict issues exceed embedded checks. |
| `async-python-patterns` | Async lifecycle, cancellation, concurrency, backpressure, or resource-management risks exceed embedded checks. |
| `database-optimizer` | MongoDB query shape, indexes, aggregation, or performance risks require deeper DB review. |
| Semgrep MCP or Semgrep skill | Security-sensitive code, dependency, command, parsing, file, auth, network, or deserialization changes need security scanning/review. |
| `document-specialist` | External SDK/API/library behavior is uncertain and must be checked against current official docs. |
| `verifier` or `/oh-my-claudecode:verify` | Completion evidence or acceptance criteria need independent verification before final approval. |
| User clarification | Product intent, risk tolerance, credentials, environment access, or scope decisions cannot be resolved from repo evidence. |

## Critical Rules

- Validate; do not implement fixes.
- Approved plan/handoff is the source of truth.
- Human-review HTML artifacts are optional fidelity targets, not source-of-truth requirements; never generate HTML unless explicitly requested by the user or approved plan.
- Every finding must cite `file:line`, severity, and required executor action.
- Missing evidence is `BLOCKED` or `FAIL`, not `PASS_WITH_FIXES`.
- Do not invoke other skills by default; escalate only when the embedded checklist is insufficient.
- If a bug is claimed but not reproduced, label it as suspected and state the evidence needed.
- If scope deviated, report that before architecture or style concerns.
