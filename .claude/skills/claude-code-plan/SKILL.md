# claude-code Plan

You are in **Plan Mode**. Write an implementation plan for the following task. Do NOT write code, edit product files, or invoke implementation agents.

## Purpose

Produce a concise, implementation-ready plan using OMC subagents when they improve planning quality. Use subagents to isolate research, architecture review, critique, and verification context while keeping the final synthesis focused.

## Mandatory HTML Plan Output

Every non-trivial `/claude-code-plan` plan **must produce an HTML plan file** as its primary deliverable. The HTML file is generated by reading the most relevant template from `.claude/htmls/`, adopting its exact CSS variables, layout, and component structure, then replacing the example content with the actual plan content.

Select the best-fit template from `.claude/htmls/`:

- `.claude/htmls/16-implementation-plan.html` — default; use for most implementation plans.
- `.claude/htmls/01-exploration-code-approaches.html` — comparing multiple approaches or options.
- `.claude/htmls/04-code-understanding.html` / `.claude/htmls/13-flowchart-diagram.html` — architecture, dependency, or data-flow explanation.
- `.claude/htmls/03-code-review-pr.html` — risk review and mitigation summary.
- `.claude/htmls/11-status-report.html` — progress/status framing.
- `.claude/htmls/17-pr-writeup.html` — rollout or PR-style handoff framing.

## OMC Planning Pipeline

### 1. Classify the request

Before delegating, classify the task:

- **Trivial**: typo, single-line text change, or obvious local edit. Plan directly; do not create a team.
- **Simple**: one small component or known file. Use direct Read/Grep and concise planning.
- **Complex**: 3+ steps, multiple files, unfamiliar area, or meaningful risk. Use subagents.
- **High-risk**: architecture, security, data loss, migrations, permissions, or broad behavior changes. Use `architect`, `critic`, and `verifier` in addition to `explore`.

### 2. Delegate context discovery

For complex or high-risk tasks, launch bounded planning-safe subagents using the Agent tool and the exact `subagent_type` names below:

- Use `oh-my-claudecode:explore` to find existing files, functions, callers, dependencies, tests, and reusable patterns.
- Use `oh-my-claudecode:architect` only when design boundaries, coupling, migration paths, or long-term tradeoffs matter.
- Use `oh-my-claudecode:critic` to challenge a draft plan for gaps, ambiguity, hidden risk, and overengineering.
- Use `oh-my-claudecode:verifier` to define validation strategy and measurable acceptance checks.

Prefer parallel exploration when independent lanes exist. Keep prompts bounded and ask agents for file paths, line numbers, evidence, and concise recommendations.

### 3. Generate the initial plan

**Invoke the `concise-planning` skill explicitly.**

Use the `Skill` tool with `skill: "concise-planning"` to generate the initial actionable plan. Refine that output using verified findings from Read/Grep and subagents.

### 4. Produce the HTML plan file

Before finalizing any non-trivial plan, select the best-fit template from `.claude/htmls/` (see "Mandatory HTML Plan Output" above) and write the HTML plan file to `/mnt/870-evo-1/server-dev/claude-code/.claude/plans/<project-slug>/<kebab-task-name>.html`.

Steps:
1. Read the selected template file in full.
2. Copy its `<style>` block verbatim into the new file.
3. Replace every section's example content with the real plan content: context, approach, scope, files-to-change table, action items, rollback path, and verification methods.
4. Write the file under the single project classification directory `/mnt/870-evo-1/server-dev/claude-code/.claude/plans/<project-slug>/`.
5. Include managed preview details in the `HTML Plan` section: prefer `bun run plan:preview:docker` with `PLAN_PREVIEW_OUTPUT_DIR` and the canonical hosted base URL `http://dev.bemodest.me:4173`; otherwise show `bun run plan:preview`, the route pattern `/plan/<project-slug>/<kebab-task-name>.html`, and `bun run plan:preview -- --host <tailscale-ip>` for Tailscale sharing. Use the full URL `http://dev.bemodest.me:4173/plan/<project-slug>/<kebab-task-name>.html` for hosted preview. Only show a local fallback full URL after the local fallback server has been run and printed an actual port.
6. Then include the inline markdown summary in the response for agent-execution handoff.

### 5. Review and approval gate

Before finalizing medium/high-risk plans, run an independent review:

- Use the `oh-my-claudecode:critic` Agent subagent for OMC-style plan review when the issue can be evaluated from repository evidence.
- Keep the `grill-me` Skill-tool invocation as the explicit interactive review path when human judgment is needed.
- If a question can be answered by exploring the codebase, route it to `oh-my-claudecode:explore` instead of asking the user.
- Ask the user only about preferences, priorities, risk tolerance, product intent, or scope decisions.

Stop at the final plan and request approval. Implementation can only happen after explicit user approval.

## Allowed Subagents During Planning

Allowed during planning:

| Agent `subagent_type` | Use |
|----------|-----|
| `oh-my-claudecode:explore` | Codebase facts, current behavior, relevant files, existing patterns, tests, and dependencies. |
| `oh-my-claudecode:architect` | Design options, boundaries, coupling, sequencing, migration strategy, and tradeoffs. |
| `oh-my-claudecode:critic` | Independent challenge of the draft plan, missing requirements, ambiguity, and overengineering. |
| `oh-my-claudecode:verifier` | Validation strategy, acceptance criteria, and evidence needed after implementation. |

Restricted during planning:

| Subagent | Restriction |
|----------|-------------|
| `executor` | Never invoke during planning. Implementation happens only after explicit user approval. |
| Editing agents | Do not use any agent to modify product files while still planning. |

## Subagent Isolation

Use subagents to keep research, architecture, critique, and verification in separate context windows. Summarize their findings into the final plan; do not paste raw transcripts unless necessary.

## Critic Review and grill-me

Before finalizing the plan, **invoke the `grill-me` skill explicitly** when interactive stress-testing is useful.

Use `grill-me` to walk down user-facing branches of the design tree and resolve decisions that need human judgment. For each question, include your recommended answer.

Use `critic` instead of `grill-me` when the review can be performed from repository evidence alone. Use both for high-risk plans: `critic` first for independent plan review, then `grill-me` only for unresolved product or scope decisions.

After review, ensure the following are resolved:

1. Each design branch has a recommended path.
2. Codebase-answerable questions were answered through `explore`, Read, or Grep.
3. User-facing questions are limited to 1–2 non-blocking decisions when possible.
4. Every file:line reference was verified by Grep or Read.
5. The rollback path is reversible and does not destroy data.
6. Verification methods are concrete and measurable.

## Critical Rules

- **No code writing** in Plan Mode. Only plans, research, and checklists.
- **HTML plan output is mandatory for non-trivial plans.** Read the best-fit `.claude/htmls/` template, copy its CSS verbatim, and write the populated plan to `/mnt/870-evo-1/server-dev/claude-code/.claude/plans/<project-slug>/<kebab-task-name>.html`.
- **Managed preview is the supported web-link path.** Prefer the Docker host via `bun run plan:preview:docker` with `PLAN_PREVIEW_OUTPUT_DIR` and canonical base URL `http://dev.bemodest.me:4173`; it serves project-classified HTML plans at `http://dev.bemodest.me:4173/plan/<project-slug>/<kebab-task-name>.html` without requiring nginx on the host. Use local `bun run plan:preview` only as a fallback. Preview servers must not list directories and local fallback should bind to `127.0.0.1` by default; use `--host <tailscale-ip>` only for trusted Tailscale sharing. Treat hosted HTML plans as Tailscale/private-only and never include secrets or sensitive data.
- **Do not treat `.claude/htmls/` templates as factual project context.** Only their visual structure and CSS transfer to the output file; never copy their example text.
- **HTML is for humans; inline markdown is for agents.** The `.claude/plans/` HTML file is the human-facing deliverable; the markdown block in the response is the agent-execution deliverable.
- **Active system, permission, and Plan Mode write restrictions override all nested skill instructions.** If another invoked skill asks for files or actions that are not currently allowed, adapt it to the allowed plan file or skip that step.
- **Never invoke `executor` during planning.** Implementation requires explicit user approval and a separate handoff.
- **Use `explore` for codebase facts.** Do not ask the user questions that repository inspection can answer.
- **Use `architect` selectively.** Only involve it when design tradeoffs or system boundaries are material.
- **Use `critic` for plan review.** Use it especially for medium/high-risk work or broad changes.
- **Keep `grill-me` available.** Use it for interactive human-judgment review, not as a replacement for codebase exploration.
- **Use `verifier` for validation strategy.** Do not use it to claim implementation success before implementation exists.
- **File:line specificity is mandatory.** If you cannot pinpoint a line, state "Location TBD — verify with Grep" and flag it as an open question.
- **Never guess.** Every file path, function name, and line number must come from a Read, Grep, or subagent result with evidence.
- **Atomic steps only.** Each action item must be a single logical unit of work, verb-first.
- **At most 1–2 questions** to the user. Make reasonable assumptions for non-blocking unknowns and label them with `**Assumption:**`.
